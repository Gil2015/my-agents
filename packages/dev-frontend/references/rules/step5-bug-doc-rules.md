# 缺陷文档填写规则

`.ai/missions/{missionId}/bugDocs/bug.md` 在修复阶段的填写与增量更新规则。主模板以 `../doc-templates/bug-doc-template.md` 为准，本文件补充字段约束、状态约束和回归闭环规则。

## 顶部摘要填写规则

| 字段 | 填写方式 | 说明 |
|------|---------|------|
| 模块名 | 与实际模块目录一致 | 不写页面标题或口语简称 |
| 模块显示名 | 可选，优先沿用 `config.json.module.displayName` | 只用于文档展示，不参与代码路径解析 |
| 当前结论 | 1 句摘要 | 例如“BUG-003 已修复，仍有 1 个 OPEN 待处理” |
| 修复进度 | `OPEN x / FIXING y / FIXED z / BLOCKED n / WONT_FIX m` | 必须由当前条目状态统计得出 |
| 本轮来源 | 沿用或补充来源摘要 | 不要抹掉第四步收集到的来源信息 |
| 优先处理 | `BUG-*` 列表或 `无` | 写当前还需要优先处理的条目 |

## 阶段职责边界

- 第五步只处理已经存在的 `BUG-*`
- 第五步不新增 `BUG-*`
- 如果回归时发现独立新问题，应在当前条目的 `剩余风险` 或 `回归说明` 中如实记录，并回到 `module-test` 重新建档
- 不要把无关新问题硬并进当前 Bug，伪装成“同一个问题”

## `BUG-*` 编号规则

- 每个缺陷使用唯一 `BUG-*` 编号
- 第五步只更新已有编号，不在这一阶段新增编号
- 不要因为修复完成就删除旧条目

## 单条缺陷字段规则

| 字段 | 填写方式 | 说明 |
|------|---------|------|
| 标题 | 一句话说明问题 | 要能直接看出故障点 |
| 严重级别 | `S0` / `S1` / `S2` / `S3` | 与真实影响匹配 |
| 来源 | `用户反馈` / `bugDocSources` / `AI审查` / `监控` 等 | 保留真实来源 |
| 来源文件 | 真实路径或 `无` | 第四步已记录时保留，不要抹掉 |
| 状态 | `OPEN` / `FIXING` / `FIXED` / `BLOCKED` / `WONT_FIX` | 与当前进度一致 |
| 关联需求 | `REQ-*` / `AC-*` / `USER_INPUT` / 留空 | 无明确需求时不要伪造 |
| 关联代码 | 模块内真实文件路径或链路描述 | 修复阶段应尽量补齐 |
| 关联接口 | `API-*` / 接口标识 / `无` | 没有接口依赖时写 `无` |
| 复现环境 | 浏览器、账号、环境、mock 条件等 | 方便他人复现 |
| 剩余风险 | 1 句说明 | 写未覆盖范围、相邻风险或阻塞项 |
| 是否关闭 | `是` / `否` | 与状态同步 |

## 严重级别定义

| 级别 | 含义 | 示例 |
|------|------|------|
| S0 | 阻塞发布或存在高风险 | 页面崩溃、数据损坏、安全问题 |
| S1 | 核心链路不可用 | 无法提交、保存、删除、支付 |
| S2 | 功能可用但行为错误 | 格式错、状态错、结果不一致 |
| S3 | 轻微外观或体验问题 | 对齐异常、文案问题、轻微样式瑕疵 |

## 状态与关闭规则

| 状态 | 含义 | 是否关闭 |
|------|------|---------|
| OPEN | 已记录，尚未开始修复 | 否 |
| FIXING | 正在修复或等待本轮回归 | 否 |
| FIXED | 修复完成且回归通过 | 是 |
| BLOCKED | 受外部依赖阻塞 | 否 |
| WONT_FIX | 明确决定不修 | 是 |

规则：
- 没有回归结果，不要写 `FIXED`
- 明确还需继续处理时，不要提前关闭
- `WONT_FIX` 必须在 `### 回归结果` 或 `剩余风险` 中说明原因

## 内容填写要求

### `问题描述`

- 保留问题症状和用户感知
- 不要在修复阶段把第四步收集到的原始问题语义抹掉

### `复现步骤`

- 必须能被他人复现
- 包含页面入口、账号/数据条件、操作步骤
- 如果依赖 mock、灰度环境或特定角色，必须写清楚

### `期望结果`

- 直接描述正确行为
- 必须可判断通过或失败
- 不写主观话术，例如“体验良好”

### `实际结果`

- 记录当前真实现象
- 可包含错误提示、控制台信息、接口异常摘要
- 不能只写“有问题”或“失败”

### `根因分析`

- 必须在动代码前填写
- 解释为什么发生，而不是只描述表象
- 尽量写明文件、函数、字段、条件分支或时序问题

### `修复方案`

- 说明准备修改什么
- 如果只是临时规避，也要明确写出，不要伪装成最终方案

### `回归结果`

- `回归状态` 写 `PASS` / `FAIL` / `BLOCKED` / `PENDING`
- `回归说明` 记录重跑了哪些命令、哪些路径、结论是什么
- 回归失败时，缺陷状态不得写 `FIXED`

## 增量更新规则

对已有 `bug.md` 重新执行时：

1. 先读取旧文档，不要整份覆盖
2. 已存在的 `BUG-*` 保留编号和历史内容
3. 本轮修复只更新目标 Bug 的相关字段，不抹掉历史上下文
4. 新发现的独立问题不要在第五步新增条目，应回到第四步重新收口
5. 每轮回归都更新 `### 回归结果`，而不是只改顶部结论
6. 顶部 `修复进度`、`当前结论`、`优先处理` 必须同步更新

## 常见错误

| 错误做法 | 正确做法 |
|---------|---------|
| 只在聊天里说“这个已经修了” | 把状态和回归结果写回 `bug.md` |
| 还没分析就先改代码 | 先写 `### 根因分析` |
| 修完后不更新顶部进度摘要 | 至少把 `bug.md` 顶部进度和条目状态同步到真实结果 |
| 把多个无关问题合并成一个 Bug | 每个独立问题单独建条目 |
| 修复失败后仍保留 `FIXED` | 改回 `FIXING` 或 `OPEN`，记录真实结果 |
