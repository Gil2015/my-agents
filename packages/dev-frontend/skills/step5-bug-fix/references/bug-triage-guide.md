# Bug 分析指南

如何分析、分类和排序前端模块缺陷，并把根因结论写回 `bug.md`。

## 从症状到根因

排查一个 Bug 时，不要从“我猜这里可能有问题”开始，而要从可观察症状逆向追踪：

1. 先确认复现条件
2. 再确认预期行为
3. 然后沿代码路径找到第一个出错环节
4. 最后写出根因和修复策略

如果你还没找到“第一个出错环节”，说明还在调查阶段，不应该进入修复阶段。

## Bug 分类

### 分类 1：数据类 Bug

数据错误、缺失或格式不正确。

**表现症状：**
- 显示值错误
- 页面出现 `Invalid Date`、`NaN`、`undefined`
- 应有数据的字段为空
- 展示数据与接口响应不一致

**排查路径：**

```text
Layout / Component <- hooks/index.ts <- useData <- service <- API response
```

**常见原因：**
- 接口响应与 `defs/type.ts` 定义不一致
- `useData` 中缺少数据映射或空值兜底
- 响应字段名和展示字段名混用
- 接口包装层级解析错误

### 分类 2：交互类 Bug

用户操作未产生预期结果。

**表现症状：**
- 点击无反应
- 触发了错误动作
- 操作成功但 UI 未更新
- 一次点击执行了两次

**排查路径：**

```text
UI event -> Layout handler -> useController -> service / state update -> re-render
```

**常见原因：**
- 事件未正确绑定
- `useController` 闭包使用了过期数据
- 操作成功后缺少刷新或状态同步
- 条件判断分支写反

### 分类 3：状态与副作用 Bug

状态进入无效、过期或互相覆盖的状态。

**表现症状：**
- 切页后仍显示旧数据
- Loading 卡住不消失
- 分页、筛选或表单状态被意外重置
- 页面刷新或返回后状态异常

**排查路径：**

```text
Current state -> useData / useSetState -> useWatcher / request trigger -> render result
```

**常见原因：**
- 状态初始化或重置不完整
- 请求触发时机错误
- 依赖项缺失，导致副作用没有更新
- 多个状态更新互相覆盖

### 分类 4：布局与样式 Bug

视觉渲染、布局层级或响应式表现不正确。

**表现症状：**
- 对齐错位
- 元素溢出、遮挡、裁剪
- 响应式切换后布局崩溃
- 文本截断、按钮位置异常

**排查路径：**

```text
CSS Module -> classNames -> container layout -> UI library style override
```

**常见原因：**
- CSS Module 类名未正确引用
- 父容器布局属性缺失
- 组件结构变化后样式未同步调整
- UI 库默认样式覆盖了本地样式

### 分类 5：类型与契约 Bug

类型定义和真实数据契约不一致。

**表现症状：**
- TypeScript 编译错误
- 运行时报 `Cannot read properties of undefined`
- 自动补全和真实字段不一致
- 大量 `as any` 或断言绕过了真实问题

**排查路径：**

```text
defs/type.ts -> defs/service.ts -> hooks -> layouts / components
```

**常见原因：**
- 接口字段可空性写错
- 参数和响应类型没有跟文档同步
- 展示层衍生字段混进了接口类型
- 类型链断裂，导致上下游约束失效

## 严重级别建议

与 `bug.md` 中的 `严重级别` 保持一致：

| 级别 | 含义 | 示例 |
|------|------|------|
| S0 | 阻塞发布或存在高风险 | 页面崩溃、数据损坏、安全问题 |
| S1 | 核心路径受损 | 无法创建、提交、删除、保存 |
| S2 | 功能可用但行为错误 | 格式错误、状态错误、异常提示不对 |
| S3 | 轻微外观或体验问题 | 对齐偏差、文案问题、轻微样式异常 |

## 根因分析写法

将分析结果沉淀到 `bug.md` 的 `### 根因分析` 中，至少回答这四个问题：

1. 症状是什么
2. 错误发生在代码链路的哪一层
3. 真正原因是什么
4. 修复要改哪一层，为什么

推荐写法：

```markdown
### 根因分析

- 症状：列表删除成功后 UI 未刷新
- 代码路径：Layout 删除按钮 -> useController.handleDelete -> services.deleteItem -> useData.list
- 根因：删除成功后只弹了成功提示，没有重新触发列表查询，因此页面仍保留旧数据
- 修复策略：在 `handleDelete` 成功回调中补充刷新列表动作，并补回归测试
- 风险提示：同类新增/编辑成功后的刷新逻辑需要一起回归
```

坏的写法：
- “删除有问题”
- “接口返回不对，先兜底一下”
- “可能是 hooks 没更新”

这些都不是根因，只是模糊怀疑。

## 多 Bug 处理策略

当存在多个 Bug 时：

1. 先按严重级别排序：`S0 -> S1 -> S2 -> S3`
2. 再看是否存在依赖关系：前一个修复是否会顺带解决后一个
3. 逐个完成“分析 -> 修复 -> 回归 -> 更新文档”的完整闭环
4. 不要同时推进多个互相干扰的半成品修复

## 回归范围判断

一个 Bug 修完后，至少回归以下范围：

- 该 Bug 直接关联的 `TC-*` / `MC-*`
- 同一 `REQ` 下经过相同代码路径的其他测试点
- 相同数据结构、相同 hooks 或相同布局分支下的关键场景
- 根因分析中明确提到的边界场景

## 需要避免的反模式

| 反模式 | 为什么有害 | 正确做法 |
|------|-----------|---------|
| 散弹式调试 | 随机改代码，难以定位真实原因 | 按代码链路逐层排查 |
| 复制粘贴式修复 | 容易制造重复逻辑和新的分叉 | 在根因所在处修复 |
| 防御性过度工程 | 到处加空值判断会掩盖问题来源 | 找出为什么为空 |
| 静默吞掉异常 | 错误看似消失，实际只是隐藏 | 显式处理并记录真实错误 |
| “我本地没问题” | 忽略环境差异和复现条件 | 把复现环境写进 `bug.md` |
