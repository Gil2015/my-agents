# 缺陷文档填写规则

`.ai/missions/{module}/bugDoc/bug.md` 的填写与增量更新规则。主模板以 `../../references/bug-doc-template.md` 为准，本文件补充字段约束、状态约束和回归闭环规则。

## 顶部摘要填写规则

| 字段 | 填写方式 | 说明 |
|------|---------|------|
| 模块名 | 与实际模块目录一致 | 不写页面标题或口语简称 |
| 当前结论 | 1 句摘要 | 例如“仍有 2 个 OPEN，需继续回归” |

## `BUG-*` 编号规则

- 每个缺陷使用唯一 `BUG-*` 编号
- 同一文档内编号递增，不复用旧编号
- 已有缺陷保留原编号，新增缺陷使用下一个可用编号
- 不要因为修复完成就删除旧条目

## 单条缺陷字段规则

| 字段 | 填写方式 | 说明 |
|------|---------|------|
| 标题 | 一句话说明问题 | 要能直接看出故障点 |
| 严重级别 | `S0` / `S1` / `S2` / `S3` | 与真实影响匹配 |
| 来源 | `测试` / `开发者` / `用户反馈` / `监控` 等 | 保留真实来源 |
| 状态 | `OPEN` / `FIXING` / `FIXED` / `BLOCKED` / `WONT_FIX` | 与当前进度一致 |
| 关联需求 | `REQ-*` / `USER_INPUT` / 留空 | 无明确需求时不要伪造 |
| 关联测试 | `TC-*` / `MC-*` / `USER_INPUT` / 留空 | 尽量关联到具体测试点 |
| 关联接口 | `API-*` / 接口标识 / `无` | 没有接口依赖时写 `无` |
| 复现环境 | 浏览器、账号、环境、mock 条件等 | 方便他人复现 |
| 剩余风险 | 1 句说明 | 写未覆盖范围、相邻风险或阻塞项 |
| 是否关闭 | `是` / `否` | 与状态同步 |

## 严重级别定义

| 级别 | 含义 | 示例 |
|------|------|------|
| S0 | 阻塞发布或存在高风险 | 页面崩溃、数据损坏、安全问题 |
| S1 | 核心链路不可用 | 无法提交、保存、删除、支付 |
| S2 | 功能可用但行为错误 | 格式错、状态错、结果不一致 |
| S3 | 轻微外观或体验问题 | 对齐异常、文案问题、轻微样式瑕疵 |

## 状态与关闭规则

| 状态 | 含义 | 是否关闭 |
|------|------|---------|
| OPEN | 已记录，尚未开始修复 | 否 |
| FIXING | 正在修复或等待本轮回归 | 否 |
| FIXED | 修复完成且回归通过 | 是 |
| BLOCKED | 受外部依赖阻塞 | 否 |
| WONT_FIX | 明确决定不修 | 是 |

规则：
- 没有回归结果，不要写 `FIXED`
- 明确还需继续处理时，不要提前关闭
- `WONT_FIX` 必须在 `### 回归结果` 或 `剩余风险` 中说明原因

## 内容填写要求

### `复现步骤`

- 必须能被他人复现
- 包含页面入口、账号/数据条件、操作步骤
- 如果依赖 mock、灰度环境或特定角色，必须写清楚

### `期望结果`

- 直接描述正确行为
- 必须可判断通过或失败
- 不写主观话术，例如“体验良好”

### `实际结果`

- 记录当前真实现象
- 可包含错误提示、控制台信息、接口异常摘要
- 不能只写“有问题”或“失败”

### `根因分析`

- 必须在动代码前填写
- 解释为什么发生，而不是只描述表象
- 尽量写明文件、函数、字段、条件分支或时序问题

### `修复方案`

- 说明准备修改什么
- 如果只是临时规避，也要明确写出，不要伪装成最终方案

### `回归结果`

- `回归状态` 写 `PASS` / `FAIL` / `BLOCKED` / `PENDING`
- `回归说明` 记录重跑了哪些 `TC-*` / `MC-*`，结论是什么
- 回归失败时，缺陷状态不得写 `FIXED`

## 与 `testDocs/test.md` 的联动规则

- 关联测试存在时，修复后必须同步更新对应测试条目
- 测试条目的 `状态`、`实际结果` 和 Bug 关联要与 `bug.md` 一致
- 如果缺陷暴露出测试覆盖缺口，应补上最小必要的回归测试条目

## 增量更新规则

对已有 `bug.md` 重新执行时：

1. 先读取旧文档，不要整份覆盖
2. 已存在的 `BUG-*` 保留编号和历史内容
3. 本轮修复只更新相关字段，不抹掉历史上下文
4. 新发现的问题新增条目，不要塞进旧 Bug 里混写
5. 每轮回归都更新 `### 回归结果`，而不是只改顶部结论

## 常见错误

| 错误做法 | 正确做法 |
|---------|---------|
| 只在聊天里说“这个已经修了” | 把状态和回归结果写回 `bug.md` |
| 还没分析就先改代码 | 先写 `### 根因分析` |
| 修完后不更新 `test.md` | 同步更新关联测试条目 |
| 把多个无关问题合并成一个 Bug | 每个独立问题单独建条目 |
| 修复失败后仍保留 `FIXED` | 改回 `FIXING` 或 `OPEN`，记录真实结果 |
