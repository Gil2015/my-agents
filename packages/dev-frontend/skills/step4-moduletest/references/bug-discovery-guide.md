# 问题发现与审查指南

如何把用户口述问题、`bugDocSources` 来源文件和 `req.md -> 代码` 审查结果，整理成可进入修复阶段的 `BUG-*` 条目。

## 来源优先级

默认按以下顺序处理：

1. 用户或开发者当前明确提到的问题
2. `config.json` 中 `bugDocSources` 指向的外部材料
3. 根据 `.ai/missions/{missionId}/reqDocs/req.md` 对代码做静态审查发现的问题

原因很简单：显式问题优先级最高，源文件问题次之，AI 审查补盲区。

## `bugDocSources` 读取规则

- 路径必须来自 `.ai/missions/{missionId}/config.json`
- 文件和目录都可作为来源
- 相对路径优先按 `projectRoot` 解析
- 路径不存在时，不要静默忽略；要在 `bug.md` 顶部结论或阻塞说明中写明
- 来自来源文件的问题，尽量在条目中保留真实文件路径

## `req.md -> 代码` 审查方法

默认采用“需求映射 + 代码链路 + 边界清单”的三层审查法。

### 第一层：需求映射

把 `REQ/AC` 映射到真实实现：

- 页面入口在哪个模块
- 哪个布局负责展示
- 哪个 hook 或 controller 负责交互
- 哪个 service 负责数据获取或提交
- 哪个状态分支负责空态、错误态、权限态

常见问题：
- `REQ` 在代码中根本找不到实现入口
- `AC` 要求的行为存在，但触发条件写反了
- 需求要求刷新列表，代码只提示成功不刷新
- 需求要求空态占位，代码默认白屏或渲染异常

### 第二层：代码链路审查

沿代码链路追踪到最容易出错的位置：

```text
UI -> Layout -> hooks/useController -> hooks/useData -> service -> type -> render
```

重点检查：
- 事件有没有绑上
- 参数有没有传对
- 成功后状态有没有回写
- 失败后错误有没有落到 UI
- 路由跳转和返回是否处理一致

### 第三层：边界与异常清单

不是每个模块都要全做一遍，但必须有意识地筛查：

#### 数据边界

- 空列表 / 空对象 / 空字段
- `null` / `undefined`
- 超长文本、特殊字符
- 接口返回字段缺失、字段重命名、类型不一致

#### 状态边界

- 首次加载
- 加载中重复点击
- 成功后刷新
- 失败后恢复
- 权限不足
- 会话过期

#### 交互边界

- 双击提交
- 连续切换分页、筛选或标签
- 弹窗确认 / 取消 / 关闭链路
- 回退、前进、刷新页面

#### 视觉与布局风险

- 溢出、遮挡、滚动异常
- 表格列截断、按钮错位
- 响应式断点表现
- UI 库默认样式覆盖本地样式

## 什么时候算 Bug，什么时候不算

应该记为 Bug：
- 已有需求明确说明了预期，代码未实现或实现错误
- 用户反馈的问题能明确对应到当前模块行为
- 来源文件或代码审查能指出“现在实际是什么”和“应该是什么”

不应直接记为 Bug：
- 需求本身还没定
- 只是“看起来可能有点怪”，但没有行为依据
- 完全无法定位模块、页面入口或触发步骤

这类情况应回到 `req-collect` 或先补上下文。

## AI 审查的证据标准

静态审查发现的问题也可以入档，但要满足以下条件：

- 能指出违反的是哪条 `REQ/AC` 或哪段显式用户描述
- 能指出相关代码链路、文件或条件分支
- `实际结果` 里能写出清晰的代码证据，而不是“我猜”

推荐写法：

```markdown
### 实际结果

`useController.handleDelete` 成功后只调用了成功提示，没有重新触发列表查询，
与 REQ-002 / AC-001 要求的“删除后即时刷新”不一致。
```

## 严重级别建议

| 级别 | 含义 | 示例 |
|------|------|------|
| S0 | 阻塞发布或高风险 | 页面崩溃、数据损坏、安全问题 |
| S1 | 核心路径不可用 | 无法创建、保存、删除、提交 |
| S2 | 功能可用但结果错误 | 状态错、数据展示错、提示错 |
| S3 | 外观或体验问题 | 对齐异常、轻微样式问题、文案问题 |

## 收口完成标准

满足以下条件，才算可以进入 `bug-fix`：

- 显式来源已全部收口
- 每个 `BUG-*` 都有来源、期望结果、实际结果
- 顶部 `修复进度` 与条目状态一致
- 优先处理项已经给出
- 需求不明确的问题没有被伪装成 Bug
