# 测试文档填写规则

`.ai/missions/{module}/testDocs/test.md` 的填写与更新规则。主模板以 `../../references/test-doc-template.md` 为准，本文件只补充落地约束。

## 顶部摘要填写规则

| 字段 | 填写方式 | 说明 |
|------|---------|------|
| 模块名 | 与实际模块目录一致 | 不要写页面标题或中文昵称 |
| 测试范围 | 写明本轮来源 | 例如 `reqDocs/req.md + USER_INPUT + bugDoc/bug.md` |
| 总状态 | `PENDING` / `PASS` / `FAIL` / `BLOCKED` | 由条目状态汇总得出 |
| 风险摘要 | 1-3 句 | 说明未覆盖风险、阻塞项、已转 Bug |

## 用例编号规则

- 自动化用例使用 `TC-001`、`TC-002` ...
- 手动用例使用 `MC-001`、`MC-002` ...
- 同一份文档内，编号单调递增，不复用已删除编号
- 回归时优先保留已有编号；只有新增场景才分配新 ID

## 单条用例填写规则

| 字段 | 填写方式 | 说明 |
|------|---------|------|
| 关联需求 | `REQ-*` 或 `USER_INPUT` | 来自需求文档就写真实编号 |
| 关联 AC | `AC-*` 或 `USER_INPUT` | 没有 AC 时不要伪造编号 |
| 关联接口 | `API-*` / 接口标识 / `无` | 没有明确接口时写 `无` |
| 类型 | `自动` 或 `手动` | 与 `TC` / `MC` 保持一致 |
| 场景 | 一句话描述被验证的行为 | 不写“测试功能正常” |
| 前置条件 | 账号、数据、页面、mock、浏览器等 | 缺失前置条件会导致不可复测 |
| 期望结果 | 可直接判定通过或失败 | 不写主观感受 |
| 实际结果 | 记录真实执行结果与证据 | 执行后必须回填 |
| 状态 | `PENDING` / `PASS` / `FAIL` / `BLOCKED` | 不能跳过执行直接写 `PASS` |
| 已转 Bug | Bug 编号或留空 | 只有 `FAIL` 时必须补 |

## 执行步骤规则

- 每一步必须是可操作动作
- 同一用例的步骤只服务于一个明确结论
- 自动化用例也要写执行步骤，用于说明准备数据和触发方式
- 手动用例要明确页面入口、交互动作和观察点

示例：

```markdown
## TC-003

- 关联需求：REQ-002
- 关联 AC：AC-001
- 关联接口：API-002
- 类型：自动
- 场景：删除成功后列表自动刷新
- 前置条件：mock 删除接口返回成功，列表接口存在待删除数据
- 期望结果：删除成功提示出现，列表中不再显示该项
- 实际结果：执行 `npm test -- fund-list`，断言通过
- 状态：PASS
- 已转 Bug：

### 执行步骤

1. 准备包含 1 条目标数据的 mock 列表
2. 触发删除动作
3. 断言成功提示可见且目标数据已移除
```

## 总状态计算规则

按以下优先级汇总顶部 `总状态`：

1. 只要存在任意 `FAIL`，总状态就是 `FAIL`
2. 没有 `FAIL`，但存在 `BLOCKED`，总状态就是 `BLOCKED`
3. 没有 `FAIL` 或 `BLOCKED`，但仍有关键 `PENDING`，总状态也应为 `BLOCKED`
4. 所有用例都是 `PASS` 时，才能写 `PASS`
5. 文档刚生成但尚未执行时，才允许整体保持 `PENDING`

## Bug 关联规则

- 每个 `FAIL` 用例都要在 `.ai/missions/{module}/bugDoc/bug.md` 中有对应条目
- `已转 Bug` 填写对应的 Bug 编号，如 `BUG-001`
- 同一个缺陷影响多个用例时，可以复用同一 Bug 编号，但每个用例都要明确写出
- 回归通过后，不删除 Bug 关联；只更新用例状态和 Bug 状态

## 增量更新规则

对已有 `test.md` 重新执行时：

1. 先读取旧文档，不要整份覆盖
2. 仍然有效的旧用例保留原编号
3. 新增范围使用下一个可用编号
4. 已废弃的旧用例不要直接删除，可在 `场景` 或 `实际结果` 中说明本轮不适用
5. 历史 `FAIL` / `BLOCKED` 项优先回归

## 常见错误

| 错误做法 | 正确做法 |
|---------|---------|
| 先执行，最后凭记忆补文档 | 先建 case，再执行并回写 |
| 用一个 case 覆盖多个不相关结论 | 一个 case 只验证一个核心结论 |
| 没有真实编号也硬写 `REQ-999` | 写 `USER_INPUT`，保留真实来源 |
| `实际结果` 只写“通过” | 写清执行证据、环境和现象 |
| 有失败项但没写 Bug 编号 | 补 `bugDoc/bug.md` 并关联编号 |
