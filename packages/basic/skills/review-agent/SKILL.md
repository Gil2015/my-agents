# Review Agent

审查 Agent（协调者）和 SubAgent（子智能体）定义文件的结构完整性和质量。

## 概述

本技能帮助你审查 AGENTS.md 和 SubAgent 提示词是否符合最佳实践。重点检查：结构完整性、职责边界、返回状态协议、质量关卡设计。

一个写得好的 Agent 系统能让多阶段任务可靠执行；写得差的则会导致子智能体迷失方向、上下文丢失、错误状态下继续执行。

## 适用场景

**使用：**
- 审查新编写的 AGENTS.md
- 审查 SubAgent 提示词（`prompts/*.md`）
- 评估编排设计是否合理
- 在 Agent 上线前做质量检查

**不使用：**
- 审查 SKILL.md（使用 `review-skill`）
- 审查普通代码（使用项目的代码审查流程）

## 审查流程

### 第一步：审查 AGENTS.md

检查以下章节是否存在且质量合格：

| 章节 | 必需 | 检查要点 |
|------|------|---------|
| 角色 | 是 | 具体而非泛泛，说明管理什么流程 |
| 工作流程 | 是 | 有清晰的阶段流水线（文字或流程图） |
| 子智能体表 | 是 | 每个子智能体有：名称、阶段、文件路径、职责 |
| 协调规则 | 是 | 包含：阶段流转条件、调度协议、返回状态处理 |
| 并行规则 | 视情况 | 如果有可并行的子智能体，需说明并行和汇聚条件 |
| 质量关卡 | 是 | 阶段间的验证检查，应可机器验证 |
| 沟通 | 是 | 何时向用户报告、报告什么 |

### 第二步：审查每个 SubAgent 提示词

逐个检查 `prompts/*.md`，验证六段式结构：

| 章节 | 必需 | 检查要点 |
|------|------|---------|
| 角色 | 是 | 一句话、用"你"开头、说明具体做什么 |
| 输入 | 是 | 列出所有需要的数据，子智能体不需要自己找数据 |
| 执行流程 | 是 | 编号步骤、动词开头、每步一个明确动作 |
| 输出 | 是 | 精确描述格式和位置，应机器可验证 |
| 返回状态 | 是 | 四种标准状态齐全，每种有精确触发条件 |
| 规则 | 是 | 3-7 条不可违反的约束 |

### 第三步：检查整体一致性

| 检查项 | 说明 |
|--------|------|
| 子智能体表 ↔ prompts/ | 表中列出的每个子智能体都有对应的 prompt 文件 |
| 阶段流转 ↔ 前置条件 | 每个阶段都有明确的前置条件 |
| 上游输出 ↔ 下游输入 | 上一阶段的输出格式与下一阶段的输入要求匹配 |
| 返回状态 ↔ 协调规则 | 每种返回状态在协调规则中都有对应的处理动作 |
| 质量关卡 ↔ 输出格式 | 关卡检查的内容与子智能体的输出格式一致 |

### 第四步：输出审查报告

按以下格式输出：

```markdown
## Agent Review: {agent-name}

### AGENTS.md 结构: PASS / FAIL
{缺少的章节或问题}

### SubAgent 提示词
| SubAgent | 结构 | 输入完整性 | 状态协议 | 规则 | 整体 |
|----------|------|-----------|---------|------|------|
| ... | ✅/❌ | ✅/❌ | ✅/❌ | ✅/❌ | PASS/FAIL |

### 一致性检查
| 检查项 | 结果 | 说明 |
|--------|------|------|
| ... | ✅/❌ | ... |

### 发现的问题
| 位置 | 问题 | 严重度 | 修复建议 |
|------|------|--------|---------|
| ... | ... | HIGH/MED/LOW | ... |

### 整体: PASS / PASS_WITH_WARNINGS / FAIL
```

## 速查表

### 常见问题及严重度

| 问题 | 严重度 | 原因 |
|------|--------|------|
| 缺少返回状态定义 | HIGH | 协调 Agent 无法处理异常 |
| SubAgent 输入不完整 | HIGH | 子智能体被迫自己找数据，行为不可预测 |
| 缺少质量关卡 | HIGH | 下游可能收到不合格输入 |
| 角色定义泛泛 | MED | 子智能体可能超越职责边界 |
| 缺少并行规则 | MED | 可并行的任务被串行执行，效率低 |
| 规则超过 7 条 | LOW | 职责可能不清晰，考虑拆分 |
| 缺少沟通规则 | LOW | 用户可能不知道进度 |

### 写作风格检查

- 角色定义是否具体（而非"帮助用户"）
- 执行流程是否动词开头
- 输出格式是否机器可验证
- 规则是否用否定句式（更清晰的边界）
- 整体是否解释了 why（而非只有 MUST）

## 常见错误

| 错误 | 后果 | 修复 |
|------|------|------|
| AGENTS.md 中重复 prompt 内容 | Token 浪费，维护两份 | 只在 AGENTS.md 列表，细节在 prompts/ |
| 子智能体职责重叠 | 重复工作或互相矛盾 | 明确边界，一个子智能体只做一件事 |
| 质量关卡是主观判断 | 无法自动验证，形同虚设 | 改为可检查的客观条件（文件存在、字段完整） |
| 只有 DONE 和 BLOCKED 两种状态 | 协调器无法做精细决策 | 补全四种标准状态 |
| SubAgent 的输出格式未定义 | 下游无法解析 | 给出精确的输出模板 |
