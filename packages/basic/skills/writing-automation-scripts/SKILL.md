# 编写自动化脚本

## 概述

把口述的本地自动化需求收敛成安全、可复用、可维护的脚本交付物。

这个 skill 不只面向开发流程，也适用于：
- 本机文件整理、搬运、重命名、清理
- 应用启动、切换、浏览器或 IDE 控制
- 剪贴板、通知、目录、日志等电脑操作
- 调用本地 AI CLI、HTTP API 或已有自动化工具去完成某一步

默认目标：
- 面向 macOS 本地自动化场景
- 输出到 `scripts/<task-slug>/`
- 默认交付原子步骤脚本，可按需附带 `run.command` 入口
- 必须附带使用说明文档，并显式写出假设、风险和验证方式

## 适用场景

当需求听起来像下面这些时，使用这个 skill：
- “帮我写个脚本”
- “一键启动项目”
- “同时打开几个工程并执行命令”
- “npm link + watch + dev + 打开浏览器”
- “整理下载目录或桌面文件”
- “控制浏览器、IDE 或别的桌面应用”
- “调用某个 AI CLI/API 跑一遍处理流程”
- “给脚本加参数”
- “把这一套本地流程自动化”

下面这些场景不要用这个 skill：
- CI/CD 流水线
- 远程主机初始化或基础设施自动化
- 分布式异步任务系统
- 容器编排或部署系统

## 默认工作流

1. 先复述用户目标和完成态。
2. 抽取或推断：
   - 工作目录
   - 命令顺序
   - 哪些步骤是一次性任务，哪些是长驻任务
   - 长驻任务的就绪信号
   - 参数、环境变量、副作用和危险操作
3. 只问真正阻塞正确性的问题，其余内容用“默认假设”明确写出。
4. 选择脚本类型。拿不准时读 `references/script-selection-matrix.md`。
5. 只要出现长驻任务，就读 `references/execution-patterns.md`，先定义就绪条件再继续后续步骤。
6. 确定交付物形态：
   - 如果交付物是文件夹，根目录必须新增 `readme.md`
   - 如果交付物是单个脚本，必须新增同 basename 的 `.md` 使用说明
7. 脚本注释规则必须先确定，再开始产出脚本正文。
8. 按下面的固定格式输出结果。

## 固定输出格式

按下面顺序输出：

1. 需求理解
2. 默认假设与风险
3. 脚本类型判断
4. 输出文件树
5. 配套说明文档
6. 脚本内容
7. 执行方式
8. 验证方式
9. 危险操作提示

如果某一节不需要，也要说明原因，不要直接省略。

## 交付规则

默认目录形态：

```text
scripts/<task-slug>/
  readme.md
  run.command
  steps/
    01-*.sh
    02-*.sh
  helpers/
    wait-ready.mjs
  extras/
    app-control.applescript
```

规则：
- 多步骤或多文件交付时，根目录必须有 `readme.md`。
- 单文件交付时，旁边必须新增同 basename 的 `.md`：
  - `start-dev.sh` 对应 `start-dev.md`
  - `run.command` 对应 `run.md`
- 说明文档至少包含：
  - 作用
  - 前置条件
  - 参数说明
  - 使用示例
  - 输出或日志位置
  - 停止或清理方式（如有长驻任务）
  - 风险提示
- 优先拆分为 `steps/NN-*.sh` 原子脚本，便于单独调试和重复执行。
- 只有在单入口更合适时，才额外提供 `run.command`。
- shell 脚本默认使用 `bash`，除非用户明确指定 `zsh`。
- 运行期状态写到 `/tmp/<task-slug>/` 或其他非仓库路径。
- 参数优先走 CLI，其次才是环境变量。
- 除非用户明确反对，默认保留 `--dry-run` 和 `--verbose`。

## 铁律：脚本必须有注释

所有生成的脚本都必须带注释，不能只给“能跑”的裸代码。

最低要求：
- 文件头注释：说明脚本作用、适用场景、主要入口
- 参数注释：解释关键参数、默认值或环境变量
- 关键流程注释：解释为什么这么做，尤其是等待、重试、后台化、清理、危险操作
- 非直观命令注释：说明副作用或前后依赖

不要只注释废话：
- 不要给 `cd "$DIR"` 这种显而易见的语句写低价值注释
- 要优先注释“我为什么这样做”和“改这里会影响什么”

## 选型规则

- 默认串行执行，遇错即停。
- 长驻任务没有就绪条件时，不要自动后台化。
- 就绪信号优先级：
  1. port
  2. HTTP 接口
  3. 日志关键字
  4. 保守等待后的 PID 存活
- 只有在 shell 变脆弱时才引入 Node 辅助脚本：
  - 参数解析较复杂
  - 需要结构化配置
  - 需要轮询或就绪判断
  - 需要跨步骤协调进程
- 如果核心任务是 macOS 应用控制、窗口/标签页操作、系统对话框或 Finder 行为，优先考虑 `osascript` / AppleScript。
- 如果核心任务是调用 AI 工具：
  - 优先复用用户已经安装或明确指定的 CLI / API
  - 没有现成依赖时，不要擅自引入新的收费服务或全局安装
  - 必须写清楚依赖、认证方式和失败回退策略
- `.command` 只作为 macOS 友好的单入口包装层，不要把全部复杂逻辑都堆进去。

## 必须检查的风险项

下面这些必须显式提醒并解释：
- `rm`、覆盖、移动、清理命令
- `kill`、`pkill`、进程替换
- 全局安装、全局 link、软链接变更
- 切分支、改依赖、写入 `scripts/` 之外的仓库文件
- 依赖隐式凭证或硬编码本机路径的行为

有风险时：
- 在命令前加显式警告
- 尽量用参数或环境变量控制，不要把危险行为写死
- 能回滚或清理时，要写出方式

## 常见错误

- 把所有逻辑都塞进一个巨大的 shell 文件
- `watch` 或 `dev` 直接后台启动，却没有日志、PID 和就绪规则
- 路径没加引号
- 单文件脚本没有配套说明文档
- 脚本没有关键注释，导致后续无法维护
- 简单命令链硬上 Node
- 复杂等待逻辑却硬塞进 shell

## 质量要求

生成的脚本应当满足：
- 可重复执行
- 可观测
- 可停止
- 可传参
- 可读，可由不擅长脚本的人回看
- 默认保守

相比“看起来聪明”的一行命令，更优先选择可调试、可解释的脚本交付物。
