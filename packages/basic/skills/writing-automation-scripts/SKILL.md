# 编写自动化脚本

## 概述

把口述的本地自动化需求收敛成安全、可复用、可维护的脚本交付物。

默认优先交付注释清晰、可验证、可停止的本地脚本，而不是只给一段“能跑”的命令。

## 适用场景

当需求听起来像下面这些时，使用这个 skill：
- “帮我写个脚本”
- “一键启动项目”
- “同时打开几个工程并执行命令”
- “npm link + watch + dev + 打开浏览器”
- “整理下载目录或桌面文件”
- “控制浏览器、IDE 或别的桌面应用”
- “调用某个 AI CLI/API 跑一遍处理流程”
- “给脚本加参数”
- “把这一套本地流程自动化”

## 不适用场景

下面这些场景不要用这个 skill：
- CI/CD 流水线
- 远程主机初始化或基础设施自动化
- 分布式异步任务系统
- 容器编排或部署系统

## 快速判断

| 需求特征 | 默认形态 | 额外动作 |
| --- | --- | --- |
| 单条命令或一次性任务 | 单个 `.sh` 或 `steps/*.sh` | 保留 `--dry-run` / `--verbose` |
| 多步骤或需要单入口 | `scripts/<task-slug>/` + `steps/` | 需要时补 `run.command` |
| 长驻任务或就绪判断 | `steps/` + `helpers/wait-ready.mjs` | 读取 `references/execution-patterns.md` |
| macOS 应用控制 | shell 编排 + `osascript` / AppleScript | 读取 `references/execution-patterns.md` |
| AI CLI / HTTP API 调用 | shell 编排，必要时补 Node | 读取 `references/execution-patterns.md` |

## 默认工作流

1. 先复述用户目标、完成态和交付位置。
2. 抽取或推断：
   - 工作目录
   - 命令顺序
   - 每一步属于哪类命令：一次性任务、长驻任务、交互式任务、半交互式任务
   - 长驻任务或半交互式任务的就绪信号
   - 哪些步骤依赖人工确认、菜单选择器、工作区目录选择器或系统弹窗
   - 是否涉及“调用方应用”和“目标应用”不是同一个前台应用的场景
   - 是否存在多窗口、多实例、多根工作区或需要先选中目标窗口/目标目录的场景
   - 参数、环境变量、副作用和危险操作
3. 只问真正阻塞正确性的问题，其余内容用“默认假设”明确写出。
4. 选择脚本类型。拿不准时读 `references/script-selection-matrix.md`。
5. 只要出现长驻任务、桌面应用控制或 AI 工具调用，就读 `references/execution-patterns.md`。
6. 需要触发示例时，再读 `references/example-prompts.md`。
7. 脚本注释规则必须先确定，再开始产出脚本正文。
8. 按“交付模式”执行，不要一边写文件一边把完整文件重复贴进回复。

## 交付模式

### 直接落地模式

当前环境可以写文件，且用户要的是可用脚本时：

- 直接创建或修改目标文件
- 回复里只总结：
  - 需求理解
  - 默认假设与风险
  - 输出路径或文件树
  - 执行方式
  - 验证方式
  - 危险操作提示

### 草案模式

只有在以下情况才不落地文件：

- 用户明确要“示例”“模板”“先别写文件”
- 当前环境不允许写文件

此时按下面顺序输出：

1. 需求理解
2. 默认假设与风险
3. 脚本类型判断
4. 输出文件树
5. 配套说明文档
6. 必要的脚本内容
7. 执行方式
8. 验证方式
9. 危险操作提示

如果某一节不需要，也要说明原因，不要直接省略。

## 交付规则

- 与当前仓库直接相关的任务，默认输出到 `scripts/<task-slug>/`
- 如果任务明显与当前仓库无关，不要机械写入 `scripts/`；先明确假设，必要时确认输出位置
- 多步骤或多文件交付时，根目录必须新增 `README.md`
- 单文件交付时，旁边必须新增同 basename 的 `.md`：
  - `start-dev.sh` 对应 `start-dev.md`
  - `run.command` 对应 `run.md`
- 说明文档至少包含：
  - 作用
  - 前置条件
  - 参数说明
  - 使用示例
  - 输出或日志位置
  - 停止或清理方式（如有长驻任务）
  - 风险提示
- 优先拆分为 `steps/NN-*.sh` 原子脚本，便于单独调试和重复执行
- 如果 `steps/NN-*.sh` 设计上允许单独执行，公共 helper 必须提供基于脚本目录的默认值解析
- 不要把关键路径、辅助脚本位置或项目目录只放在入口脚本里 export；否则单独执行 step 时会失效
- 如果桌面应用可能同时打开多个窗口，脚本必须设计显式的窗口定位策略（标题关键字、路径关键字、唯一标识等）
- 不要假设“刚 open 的窗口”或“当前前台窗口”一定就是下一步要发送快捷键的目标窗口
- 如果某一步会先弹“目录选择器”“工作区选择器”或其他中间 UI，再进入真正目标界面，这个中间选择动作必须显式建模
- 如果中间选择动作依赖目录名、菜单名、窗口名或选项文本，优先把这些值做成 CLI 参数或环境变量，而不是隐式写死在脚本深处
- 如果 UI 层级访问（例如 `application process`）在宿主里不稳定，优先考虑用应用自身 CLI 的隔离实例能力（如 `--user-data-dir`）来规避多窗口误投递
- 但隔离实例通常会改变用户体验；如果它只是兜底方案，最好做成可开关配置，不要默认强制开启
- 如果后续动作依赖全局快捷键或粘贴，脚本必须在发送前显式把目标应用激活到最前面；不要假设调用方应用会自动失焦
- 如果某个桌面动作高度依赖快捷键，优先尝试菜单动作，快捷键作为回退；同时要考虑键盘布局、输入法和 IDE 自定义快捷键导致的误触发
- 只有在单入口更合适时，才额外提供 `run.command`
- shell 脚本默认使用 `bash`，除非用户明确指定 `zsh`
- 运行期状态写到 `/tmp/<task-slug>/` 或其他非仓库路径
- 参数优先走 CLI，其次才是环境变量
- 除非用户明确反对，默认保留 `--dry-run` 和 `--verbose`

## 注释规则

所有生成的脚本都必须带注释，不能只给“能跑”的裸代码。

最低要求：
- 注释必须用中文，除必要时用英文
- 文件头注释：说明脚本作用、适用场景、主要入口
- 参数注释：解释关键参数、默认值或环境变量
- 关键流程注释：解释为什么这么做，尤其是等待、重试、后台化、清理、危险操作
- 非直观命令注释：说明副作用或前后依赖

不要只注释废话：
- 不要给 `cd "$DIR"` 这种显而易见的语句写低价值注释
- 要优先注释“我为什么这样做”和“改这里会影响什么”

## 选型规则

- 默认串行执行，遇错即停
- 长驻任务没有就绪条件时，不要自动后台化
- 就绪信号优先级：
  1. port
  2. HTTP 接口
  3. 日志关键字
  4. 保守等待后的 PID 存活
- 只有在 shell 变脆弱时才引入 Node 辅助脚本：
  - 参数解析较复杂
  - 需要结构化配置
  - 需要轮询或就绪判断
  - 需要跨步骤协调进程
- 如果核心任务是 macOS 应用控制、窗口/标签页操作、系统对话框或 Finder 行为，优先考虑 `osascript` / AppleScript
- 如果某个 IDE 或桌面应用的“打开终端/运行命令”入口在单项目与多根工作区下行为不同，先按更复杂的真实路径建模，不要默认它们等价
- 如果某个命令会出现交互式提示，默认不要只靠 `sleep` 继续进入下一步；优先改成非交互式、显式 ready 信号或人工 checkpoint
- 如果某个命令名看起来像长驻命令，但其内部脚本会继续弹选择器、确认步骤或二次操作，按“半交互式任务”处理，不要直接当普通 watch/dev
- 如果核心任务是调用 AI 工具：
  - 优先复用用户已经安装或明确指定的 CLI / API
  - 没有现成依赖时，不要擅自引入新的收费服务或全局安装
  - 必须写清楚依赖、认证方式和失败回退策略
- `.command` 只作为 macOS 友好的单入口包装层，不要把全部复杂逻辑都堆进去

- `references/script-selection-matrix.md`
- `references/execution-patterns.md`
- `references/example-prompts.md`
- `assets/templates/step.sh`
- `assets/templates/run.command`
- `assets/templates/wait-ready.mjs`

## 必须检查的风险项

下面这些必须显式提醒并解释：
- `rm`、覆盖、移动、清理命令
- `kill`、`pkill`、进程替换
- 全局安装、全局 link、软链接变更
- 切分支、改依赖、写入 `scripts/` 之外的仓库文件
- 依赖隐式凭证或硬编码本机路径的行为

有风险时：
- 在命令前加显式警告
- 尽量用参数或环境变量控制，不要把危险行为写死
- 能回滚或清理时，要写出方式

## 常见错误

- 把所有逻辑都塞进一个巨大的 shell 文件
- `watch` 或 `dev` 直接后台启动，却没有日志、PID 和就绪规则
- 路径没加引号
- step 脚本只能依赖入口脚本注入的环境变量，导致单独执行时找不到 helper 或目标路径
- 多窗口桌面应用里直接朝“当前窗口”发快捷键，结果命令落错窗口
- 调用方应用仍在最前面，导致全局快捷键和粘贴落在错误应用里
- 完全依赖默认快捷键打开桌面应用功能，但没有菜单兜底或可配置回退，导致不同键盘布局下误触发错误动作
- 多根工作区里“新建终端”先弹目录选择器，脚本却直接把命令粘贴进去，导致命令进入错误输入框
- 把表面上的 watch/dev 命令当作普通长驻任务处理，忽略其内部其实还会弹确认步骤或二次操作
- 把窗口名、菜单项、工作区目录名等关键 UI 文本硬写死且不可覆盖，导致用户一旦本地文案不同就整套脚本失效
- 交互式命令还没完成，脚本却靠固定等待进入下一步
- 单文件脚本没有配套说明文档
- 一边已经写文件，一边还在回复里重复粘贴全部文件内容
- 脚本没有关键注释，导致后续无法维护
- 简单命令链硬上 Node
- 复杂等待逻辑却硬塞进 shell

## 质量要求

生成的脚本应当满足：
- 可重复执行
- 可观测
- 可停止
- 可传参
- 可读，可由不擅长脚本的人回看
- 默认保守

相比“看起来聪明”的一行命令，更优先选择可调试、可解释的脚本交付物。
